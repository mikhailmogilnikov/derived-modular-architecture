In DMA, a layer isn't a title you hang on a folder by feel. A layer is a **predicate**: a set of conditions verifiable from the file system and import graph.

If a module matches the `features/` predicate ‚Äî it's a feature. Inbound edges from other modules appear ‚Äî time for `services/`. No need to argue rank: the graph already answered.

## Default tree

```text
src/
‚îú‚îÄ‚îÄ app/         # composition root
‚îú‚îÄ‚îÄ features/    # no inbound edges from other modules
‚îú‚îÄ‚îÄ services/    # has inbound edges + product flow
‚îî‚îÄ‚îÄ shared/      # has inbound edges + portable code
```

`services/` **may not exist** at project start. Until any module gets inbound from another module, services isn't needed ‚Äî rules that reference it simply don't fire.

## Layer predicates

| Layer | Condition |
| --- | --- |
| Composition root (`app/`, `pages/`, `routes/`) | Assembles modules; **no** module imports the composition root |
| `features/` | No inbound edges from other **modules** (only mount from composition root) |
| `services/` | Has inbound edges from modules **and** code implements a product flow |
| `shared/` | Has inbound edges **and** code can move to another product unchanged |

### services / shared boundary

This is the only place human judgment remains. Heuristic:

> Can this code live in another product without changes?

- **Yes** ‚Üí `shared/` (button, `formatCurrency`, HTTP client).
- **No** ‚Üí `services/` (cart, checkout flow, entity access rights for the product).

More ‚Äî [Services vs Shared](/deep-dive/services-vs-shared).

## Composition root

`app/`, `pages/`, `routes/` under `src/` mean the same thing: **assembly point**.

- Mounts features and services through `*/public/*`
- Mount **does not count** as an inbound edge for promotion ‚Äî a route rendering `<CheckoutPage />` doesn't turn checkout into a service
- Route files should be thin: import from `public/`, minimal logic

```tsx
// app/checkout/page.tsx ‚Äî thin shell
import { CheckoutPage } from "@/features/checkout/public/checkout-page";

export default function Page() {
  return <CheckoutPage />;
}
```

If the project has multiple composition roots, prefer `app/`; tooling picks `app` first on conflict.

## Dependency direction

```text
app/pages/routes  ‚Üí  features/*/public, services/*/public, shared
features            ‚Üí  services/*/public, shared
services            ‚Üí  services/*/public, shared   (no cycles)
shared              ‚Üí  shared
```

Forbidden:

- any import ‚Äúupward‚Äù (from shared into features, etc.);
- another module's internals ‚Äî only `*/public/*`;
- feature ‚Üí feature import (need a shared service or shared code).

### Feature does not import feature

If checkout and catalog must share behavior ‚Äî extract common logic into `services/` or a portable piece into `shared/`, don't pull imports between features.

### Promotion: feature ‚Üí service

A feature with inbound edges from **other modules** violates the predicate. Such a module moves to `services/`. Signal: rule `feature-has-inbound`.

The reverse is checked too: a service with no inbound edges from modules ‚Äî `service-no-inbound` (demote or remove).

## What tooling checks

| Rule | What it catches |
| --- | --- |
| `layer-direction` | Import in the wrong direction |
| `feature-to-feature` | Feature pulls another feature |
| `public-api` | Cross-module import bypassing `*/public/*` |
| `no-barrel` | Barrel `index` with re-exports inside a module |
| `no-cycle` | Cycle in the module graph |
| `feature-has-inbound` | Feature that other modules depend on |
| `service-no-inbound` | Service with no real consumers among modules |

Full graph audit ‚Äî `dma check`. More in [dma check](/tooling/cli/check).
