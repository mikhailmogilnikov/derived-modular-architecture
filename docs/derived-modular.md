# Derived Modular Architecture (DMA)

> Статус: **синтез, не устоявшийся стандарт**. Выведен из анализа существующих архитектур (ED, FSD, Modular Monolith, Colocation) как попытка довести DX до предела; скелет слоёв сходится с ED — отличия в эпистемологии: правила выводятся из измеренного графа и проверяются машиной, а не держатся на суждении. **Не валидирован практикой** — это лучшая гипотеза, а не проверенный опыт. Референсной реализации нет; всё собирается из существующего тулинга (ESLint, dependency-cruiser, кодмоды). Декларация — промежуточная форма: целевая форма стандарта — исполняемая спецификация (линтер + LSP + кодмоды), при которой этот документ становится её объяснением.

## Назначение

DMA — архитектура организации фронтенд-кода, спроектированная от трёх требований:

1. **Масштаб в обе стороны** — маленький проект является ранней стадией той же архитектуры, а не «другой архитектурой до миграции». Структура никогда не переписывается — она уплотняется по измеримым триггерам.
2. **Фреймворк-агностичность** — описывается только семантика модулей, слоёв и правил импортов. Ничего про state management, рендеринг, DI-контейнеры. Потенциальный стандарт — это спецификация (как ESM или `exports` в package.json), а не библиотека.
3. **Максимальный DX** — минимум вкусовых решений («куда положить файл» отвечает алгоритм; единственное оставшееся суждение — граница `services`/`shared`), ноль служебных файлов в модуле по умолчанию, ноль правил, которые держатся на памяти людей: всё, что не проверяется машиной, — не правило.

Ключевое отличие от соседних подходов: **правила не декларируются — они выводятся из файловой системы и enforced тулингом**. Слой — это положение папки. Публичность — это положение файла. Зависимости — это реальные импорты, которые измеряются, а не описываются.

## Четыре инварианта

Неизменны на любом масштабе — от пет-проекта до multi-team monorepo:

| Инвариант                          | Суть                                                                                        | Enforcement                   |
| ---------------------------------- | ------------------------------------------------------------------------------------------- | ----------------------------- |
| **Однонаправленные зависимости**   | Слои импортируют только «вниз»: `app → features → services → shared`                        | Линтер, один конфиг на проект |
| **Публичный API без barrel files** | Снаружи модуля импортируются только файлы из его `public/`, прямыми путями к файлам         | Линтер / resolver             |
| **Colocation по умолчанию**        | Код живёт максимально близко к месту использования; хук одного компонента лежит рядом с ним | Конвенция + ревью триггеров   |
| **Правило второго использования**  | Ничего не выносится «на будущее»; второе использование → подъём на уровень выше             | Измерение импортов тулингом   |

## Структура папок

### Слои — это предикаты, а не ранги

```
src/
├── app/        # composition root: роутинг, провайдеры, конфиг, связывание модулей
├── features/   # модули-листья: никто из модулей их не импортирует
├── services/   # модули с входящими рёбрами; создаётся лениво, первым промоушеном
└── shared/     # ui-kit, lib, api-клиент; не содержит продуктовых сценариев
```

Ключевое отличие от FSD/ED: принадлежность слою определяется **предикатом**, а не назначенным человеком «местом по смыслу»:

| Слой | Предикат |
|---|---|
| `app/` | Импортирует всех, не импортируем никем (композиционный корень, синглтон) |
| `features/` | Нет входящих рёбер от других модулей (монтируется только из `app`) |
| `services/` | Есть входящие рёбра от модулей **и** реализует продуктовые сценарии |
| `shared/` | Есть входящие рёбра **и** не содержит продуктовых сценариев (глобальные доменные *типы* — зафиксированное исключение: группа `domain/`) |

Честная оговорка о проверяемости: машиной проверяется первая половина предикатов — входящие рёбра, направление, ацикличность. Различие «продуктовый сценарий или нет» на границе `services`/`shared` — **последнее оставшееся суждение** в DMA. Эвристика для него: «может ли этот код без изменений жить в другом продукте?» — да → `shared`, нет → `services`. Всё остальное размещение решается фактами графа.

Почему слоёв модулей именно два, а не 4 и не 6. Ранг модуля в графе зависимостей — величина непрерывная (сервис сервиса сервиса...), но **качественный скачок в обязательствах один**: между «никто не зависит» (API можно ломать, модуль можно удалить) и «зависят» (стабильность, нельзя смотреть вверх). Ранги 1, 2, 3 внутри `services` различаются только глубиной — поведенческий контракт идентичен, поэтому глубина остаётся *измеренной* (DAG, ацикличность тулингом), а в папки не проецируется никогда. Лишние слои FSD — это ровно попытки квантовать непрерывное: `widgets`/`features` — различие по размеру, `entities`/`features` — по семантике («существительные vs глаголы»), отсюда вечные споры. Слой легален, только если по обе стороны границы — проверяемо разные обязательства. Таких границ у модулей две: «импортируют ли тебя» и «реализуешь ли продуктовые сценарии». Пятого слоя не существует не потому, что «решили остановиться», а потому что предикатов больше нет.

Следствия:

- Страницы — не слой: это `features` (листья), которые монтирует роутер из `app`. Модуль с входящими рёбрами *и* роутом (cart: страница + виджет для checkout) — `services`, без вкусовых споров: предикат отвечает однозначно.
- Widgets — композиция внутри модуля или в `app`.
- Перенос `features/cart → services/cart` при появлении входящих рёбер — не бюрократия, а принудительный момент проектирования: API, росший под одного потребителя, пересматривается под роль «от меня зависят».

### Эволюция проекта: слои появляются лениво, без переименований

Число слоёв капнуто предикатами навсегда, но папки создаются только когда предикат начинает нести информацию. Промежуточного слоя `modules/` с последующим переименованием нет: на старте у модулей нет входящих рёбер — предикат `features` выполняется тривиально, поэтому папка честно называется `features/` с первого дня.

- **Старт:** `app/` + `features/` + `shared/`. Папки `services/` не существует. Правило `feature ✗→ feature` действует с первого коммита — его видит обычный file-scoped линтер, графовый анализатор пока не нужен вообще.
- **Первый промоушен** (триггер: модуль понадобился другому модулю): создаётся `services/`, целевой модуль переезжает туда — и только тогда легализуется импорт. Переезд — не бюрократия: API, росший под одного потребителя, принудительно пересматривается под роль «от меня зависят». С этого момента графовый анализатор проверяет ацикличность внутри `services` (небольшое множество).
- **Домены** (триггер: подграф `services` стал плотным/глубоким или модули делят несколько команд): рост гасится **горизонтально, а не вертикально** — связный подграф выделяется в домен (`domains/billing/` со своими `features`/`services` внутри, рекурсивно) или в пакет monorepo. Вертикальная глубина графа никогда не порождает новых слоёв; когда она мешает людям — это сигнал резать граф поперёк (bounded context).

«Модуль» в документе — элемент `features/` или `services/`; все правила про `public/`, сегменты и стадии 0–4 действуют одинаково в обоих слоях.

### Структура `shared`

`shared` — единственный слой, где группировка **по типу кода** корректна: здесь нет фич, только горизонтальные возможности, которые потребляет весь проект. Имена групп совпадают с сегментами модулей (`ui`, `model`, `api`) — один словарь на всю архитектуру.

```
shared/
├── ui/          # ui-kit: примитивы без знания о продукте
├── lib/         # хелперы и инфраструктурные обвязки
├── api/         # транспорт: базовый клиент, интерсепторы
├── model/       # глобальное состояние-инфраструктура и конфиг
└── domain/      # опционально: глобальные бизнес-типы
```

| Группа | Что кладём | Что НЕ кладём |
|---|---|---|
| `ui/` | Атомарные компоненты: button, input, modal, layout-примитивы, иконки, типографика. Компонент не знает, *что* он отображает | Компонент, знающий о заказе/пользователе/корзине — это модуль или его внутренний `ui/` |
| `lib/` | Чистые утилиты и обвязки: даты, форматирование, i18n, notifications, storage, result-хелперы. Подпапки по задаче: `lib/date/`, `lib/i18n/` | Файл-свалку `utils.ts`; функцию, используемую одним модулем (colocation — вернуть в модуль) |
| `api/` | Базовый HTTP/WS-клиент, интерсепторы, auth-plumbing, типизация ошибок транспорта, сгенерированный клиент | Эндпоинты конкретной фичи — они в `api/` её модуля |
| `model/` | Инфраструктура состояния: store instance, query client, route-константы, env/config, feature flags | Бизнес-стейт фич — он в `model/` их модулей |
| `domain/` | Глобальные бизнес-типы, диктуемые общей моделью данных (`User`, `Money`, брендированные ID). Ни от чего не зависит | Типы одного модуля; появляется только по правилу второго использования |

Зачем это разделение, а не одна папка: у групп **разная стабильность и разные права**. `domain` не зависит ни от чего и меняется реже всех; `ui` не имеет права ходить в сеть и знать глобальный стейт — иначе ui-kit перестаёт быть переносимым. Рекомендуемые внутренние правила (тот же конфиг границ):

```
domain → (ничего)
lib    → domain
api    → lib, domain
model  → api, lib, domain
ui     → lib, domain          (✗ api, ✗ model)
```

Принципы жизни `shared`:

- **Начинается почти пустым.** Исключение — инфраструктурный bootstrap: базовый api-клиент, конфиг, первые ui-примитивы легальны с первого дня (очевидно, что их потребляет весь проект). Правило второго использования регулирует **продуктовый** код: хелперы, компоненты и типы попадают сюда только по факту второго потребителя; «пригодится» — антипаттерн.
- **Публично всё по умолчанию.** Слой существует, чтобы его потребляли; паттерн `public/` здесь не нужен. Приватность появляется на стадии 4, когда группа выезжает в пакет с `exports`.
- **Внутри групп — та же эволюция, что у модулей:** `ui/button.tsx` → `ui/button/` (компонент + стили + тесты) по мере роста.
- **`ui` и `api` — самые нестабильные жители** `shared` (известная боль, в ED названы «вечными источниками багов»): продукт растёт — ui-kit и транспорт меняются часто, задевая всех. Смягчение: на стадии 4 они первыми становятся пакетами с собственным версионированием.

### Эволюция модуля: стадии

Модуль начинается с одного файла и уплотняется по триггерам. Структура на каждой стадии — надмножество предыдущей: **миграций нет**.

**Стадия 0 — файл** (модуль только появился):

```
features/
└── checkout.tsx
```

Файл-модуль **целиком является своей публичной поверхностью** — правило «только `*/public/*`» применяется со стадии 1, когда появляются внутренности. Импорт `@/features/checkout` на этой стадии — импорт файла, а не barrel папки.

**Стадия 1 — плоский модуль** (файлов стало 2+, появился внутренний код):

```
features/checkout/
├── public/
│   └── checkout-page.tsx     # единственная входная точка
├── use-cart-total.ts          # internal
└── cart-row.tsx               # internal
```

**Стадия 2 — сегментация** (внутренних файлов ~7+):

```
features/checkout/
├── public/                    # ПЛОСКИЙ: входные точки, реализация здесь же
│   ├── checkout-page.tsx
│   ├── use-checkout.ts
│   └── checkout.types.ts
├── ui/                        # внутренние компоненты
│   ├── cart-row.tsx
│   └── promo-input.tsx
├── model/                     # состояние, логика
│   └── cart.store.ts
└── api/                       # запросы модуля
    └── checkout.api.ts
```

**Стадия 3 — расщепление** (публичная поверхность > ~7 входных точек):

```
features/
├── checkout/                  # оформление заказа
└── promo/                     # промокоды — выделены из checkout
services/
└── cart/                      # выделен из checkout; checkout его импортирует —
                               # предикат сразу селит его в services/
```

**Стадия 4 — пакеты** (несколько приложений): модули и `shared` выезжают в пакеты monorepo. Правила те же, enforcement через Nx tags / `exports` в package.json.

### Внутри модуля: зачем нужны сегменты

Сегменты — не украшение и не ритуал «как в FSD». Каждый решает конкретную проблему, и появляется только когда она возникла (стадия 2):

| Сегмент | Что решает | Содержимое |
|---|---|---|
| `public/` | Граница видимости: что можно снаружи | Входные точки (реализации), события, порты, внешние типы |
| `ui/` | Отделяет «как выглядит» от «как работает»: вид меняется без касания логики | Внутренние компоненты, вёрстка, локальные стили |
| `model/` | Единственное место, отвечающее «что модуль делает»: логика читается без раскопок по компонентам | Стейт, сторы, хуки с логикой, селекторы, машины состояний |
| `api/` | Изолирует транспорт: смена контракта бэкенда трогает один сегмент | Эндпоинты модуля, DTO, мапперы DTO → доменные типы |
| `lib/` | Локальные хелперы без утечки в `shared` | Чистые функции, используемые 2+ сегментами этого модуля |

Почему группировка по типу кода, которая запрещена на уровне проекта, легальна внутри модуля: на уровне проекта главная ось — смысл (домен), и type-based её разрушает. Внутри модуля смысл уже зафиксирован его именем — свободной осталась только ось «вид кода», и группировка по ней восстанавливает сканируемость, ничего не разрушая. Имена сегментов совпадают с группами `shared` — один словарь, **но права разные**, и это надо знать: `ui` модуля импортирует `model` (продуктовый компонент показывает продуктовый стейт), а `shared/ui` в `shared/model` ходить не может (ui-kit обязан оставаться переносимым). Совпадают имена, не правила.

Рекомендуемые правила зависимостей внутри модуля (включать при боли, не на старте):

```
public → ui, model, api, lib     # входные точки собирают модуль
ui     → model, lib              # вид знает о логике, не о транспорте
model  → api, lib                # логика дергает транспорт
api    → lib
```

Направление то же, что у слоёв проекта: вид → логика → данные. Нарушение (например, `api` импортирует `ui`) почти всегда означает, что файл лежит не в своём сегменте.

Если модуль внутри сложнее (богатая доменная логика, много инвариантов — финтех, B2B) — внутрь добавляется hexagonal-прослойка: `model/` делится на чистый домен и адаптеры. Это решение уровня одного модуля и не влияет на остальную архитектуру.

### Файловые конвенции

- **Тесты** — colocation: `cart.store.test.ts` рядом с `cart.store.ts`. Интеграционные тесты модуля — `tests/` внутри модуля; e2e всего приложения — на уровне `app` или вне `src`.
- **Стили** — рядом с компонентом (`cart-row.module.css` / co-located CSS-in-JS).
- **Типы** — по месту использования: типы одного файла живут в нём; типы сегмента — `types.ts` внутри сегмента; типы для потребителей — `public/*.types.ts`.
- **Именование** — файлы в kebab-case, имя несёт смысл без контекста папки (`use-cart-total.ts`, не `hook.ts`); `index.*` не существует в проекте вообще — это делает запрет barrel грепабельным.
- **Сторибуки, моки, фикстуры** — рядом с тем, что они описывают (`cart-row.stories.tsx`, `checkout.api.mock.ts`), по общему принципу colocation.

### Триггеры эволюции

Триггер — измеримый сигнал, а не вкус. Конкретные числа — **конфигурируемые дефолты** (спорить стоит с принципом, не с числом): важно, что порог задан заранее и один на проект. Идеал — тулинг («architecture doctor»), который сигналит сам; до его появления — чек на ревью:

| Сигнал                                       | Действие                                                      |
| -------------------------------------------- | ------------------------------------------------------------- |
| Модуль-файл получил второй файл              | Стадия 1: папка + `public/`                                   |
| Внутренних файлов ~7+                        | Стадия 2: сегменты `ui/` `model/` `api/`                      |
| В `public/` 8+ входных точек-реализаций (`*.types.ts`, `*.events.ts`, `ports.ts` не считаются) | Стадия 3: **расщепить модуль** (не сегментировать `public/`!) |
| Feature понадобился другому модулю           | Промоушен в `services/` (папка создаётся при первом переезде) + пересмотр его `public/` |
| Тип из `public/` нужен 2+ модулям            | Перенести в `shared/domain` — `import type` тоже ребро и тоже тянет промоушен |
| Появился цикл между модулями                 | Разорвать: порт потребителя + связывание в `app`, или выделить общую часть |
| Подграф `services` плотный / делится между командами | Горизонтальный сплит по доменам (`domains/*`)            |
| Хелпер в `shared` используется одним модулем | Опустить обратно в модуль (colocation)                        |
| Несколько приложений используют модуль       | Стадия 4: пакет                                               |

## Правила зависимостей

```
app      → features/*/public, services/*/public, shared
features → services/*/public, shared    (✗ features → features — локальное правило линтера)
services → services/*/public, shared    (ацикличность внутри services — графовый тулинг)
shared   → shared
✗ любые рёбра вверх; снаружи ✗→ внутренности модуля (только */public/*)
```

- Правила задаются **один раз на проект** (конфиг линтера), не по-модульно. Схема правил при росте не меняется; меняются только пути слоёв в конфиге (на стадии доменов — `domains/*/features` и т.д.). Пока `services/` не существует, правила для него просто спят.
- `import type` считается ребром наравне с обычным импортом: типовая зависимость — тоже зависимость (смена типа ломает потребителя). Тип, нужный 2+ модулям, живёт в `shared/domain` — это и есть выход без промоушена всего модуля.
- Разделение слоёв превращает большую часть проверки ацикличности в дешёвые локальные правила: `feature ✗→ feature` видит обычный file-scoped линтер; графовый анализатор нужен только внутри `services`.
- Исключения — только как enforced-override (см. «Исключения и манифесты»).

### Межмодульные связи: ацикличный граф, четыре механизма

Два «чистых» решения не выдерживают масштаба, и оба отвергнуты осознанно:

- **Полный запрет cross-import + центральный реестр контрактов в `app`** — папка `contracts/` растёт линейно с числом связей, все команды коммитят в неё (конфликты, размытое владение), а `registry.ts`, импортирующий адаптеры всех модулей, — это barrel уровня приложения: модуль, потянувший реестр, транзитивно тянет весь граф. Центральный реестр воспроизводит ту же болезнь, которую DMA запрещает на уровне модулей.
- **Свободные cross-import** — деградация в big ball of mud.

Вместо этого инвариант слабее и точнее: **граф модулей обязан быть ацикличным**. Машиной запрещены только структурные нарушения — цикл, ребро вверх, `feature → feature`; механизм связи выбирается по типу отношения, алгоритмически:

| Отношение | Механизм |
|---|---|
| Использует функциональность модуля ниже (сервиса) | Прямой импорт его `public/` |
| Уведомляет о своём событии, не зная подписчиков | Событие владельца в его `public/`; подписка — сверху вниз, иначе связывание в `app` |
| Прямой импорт создал бы цикл или ребро вверх | Порт потребителя + связывание в `app` |
| Чисто визуальная композиция модулей | Слоты/props в `app`, без кодовой зависимости |

**1. Прямой импорт — основной случай.** Большинство «межмодульных» связей на деле иерархические: фича использует стабильный поддомен (cart, session, user). Это ребро вниз: у цели появляется входящее ребро — предикат требует её промоушена в `services/`, и CI не пропустит `feature → feature`. Взамен церемонии — бесплатные go-to-definition, трассируемость, честный tree-shaking.

```ts
// features/checkout/model/use-cart-count.ts
import { useCart } from "@/services/cart/public/use-cart";
```

**2. Событие владельца.** Определение события живёт в `public/` эмиттера; подписчик импортирует токен, а не реализацию. Ребро статически видно (подписчик → файл события), реестр не нужен. Подписка из модуля легальна, только если она идёт **вниз** (feature подписан на событие сервиса). Если реакция живёт ниже эмиттера или в соседней фиче — подписку связывает `app`: оба импорта вниз, цикл невозможен.

```ts
// features/checkout/public/checkout.events.ts
export const checkoutCompleted = defineEvent<{ orderId: string }>("checkout-completed");

// app/wiring.ts — реакция ниже эмиттера, связывает app: оба импорта вниз
import { checkoutCompleted } from "@/features/checkout/public/checkout.events";
import { clearCart } from "@/services/cart/public/clear-cart";
checkoutCompleted.subscribe(clearCart);
```

**3. Порт потребителя — для разрыва цикла или ребра вверх.** Интерфейс порта объявляет тот, кому нужна зависимость, в своём `public/`; связывание — в `app` через провайдер/инъекцию. Рёбер вверх нет: `app` импортирует оба модуля (вниз), модуль получает реализацию снаружи.

```ts
// services/cart/public/ports.ts — cart объявляет, что ему нужно
export interface CheckoutPort {
  startCheckout(): void;
}
export const [CheckoutPortProvider, useCheckoutPort] = createPort<CheckoutPort>();
```

```tsx
// app/providers.tsx — связывание, оба импорта вниз
<CheckoutPortProvider value={checkoutAdapter}>{children}</CheckoutPortProvider>
```

Важная оговорка: порт — это **runtime-зависимость, невидимая в графе импортов**. Сервис фактически зависит от фичи, но анализатор этого ребра не видит — поэтому порты обязаны жить строго в `public/ports.ts` (конвенция делает их перечислимыми для тулинга), а протаскивание портом зависимости, которая легально выражается прямым импортом, — обход предиката промоушена и антипаттерн.

Следствие для масштаба: количество портов растёт с количеством **разорванных циклов**, а не с количеством связей. Много портов — сигнал не «архитектура не тянет», а «модули порезаны поперёк домена»: лечится перенарезкой, не механизмом.

## Публичный API без barrel files

### Проблема barrel files

`index.ts` с реэкспортами — стандарт де-факто (FSD, Bulletproof, ED), но:

- side effects тянут весь модуль в бандл, tree-shaking деградирует;
- TS server пережёвывает цепочки реэкспортов — медленный IDE;
- циклические зависимости через index.ts;
- HMR инвалидирует пол-модуля;
- анализаторы графа видят рёбра «через прокладку», а не реальные.

### Решение DMA

Публичность — **метаданные через размещение**, импорты — **прямые пути к файлам**:

```ts
// ✓ прямой импорт публичного файла — точное ребро графа
import { CheckoutPage } from "@/features/checkout/public/checkout-page";

// ✗ deep import внутренностей — ошибка линтера
import { CartRow } from "@/features/checkout/ui/cart-row";

// ✗ barrel — его просто не существует
import { CheckoutPage } from "@/features/checkout";
```

### Правила `public/`

1. `public/` **всегда плоский.** Контракт структурируется по потребителю, реализация — по удобству автора; это разные законы. Потребителю безразлично, где внутри лежит логика.
2. **Реализация входных точек живёт прямо в** `public/` и свободно импортирует внутренние сегменты. «Входные точки» — полноценный сегмент наравне с `ui/` и `model/`, его семантика — «контракт».
3. **Семантика — именами, не папками:** `checkout-page.tsx`, `use-checkout.ts`, `checkout.types.ts`.
4. **Разрастание** `public/` **— триггер расщепления модуля**, а не сегментации `public/`. Плоская папка физически сопротивляется распуханию контракта — боль проявляется до гниения, а не после. В лимит считаются **входные точки-реализации**; служебные файлы контракта (`*.types.ts`, `*.events.ts`, `ports.ts`) не считаются — иначе обычный сервис (страница + виджет + пара хуков + служебные) ложно срабатывал бы на расщепление.
5. Граничный случай: 1:1-реэкспорт (`public/use-checkout.ts` → `../model/use-checkout`) допустим — это симлинк, а не barrel (barrel ломает тулинг _агрегацией_ десятков экспортов, одиночный реэкспорт проходит насквозь). По умолчанию — реализация в `public/`, без прослоек.

## Исключения и манифесты

DMA почти не использует декларации, потому что метаданные, которые ничем не проверяются, врут через месяц (судьба UML и architecture description languages). Принцип: **декларация допустима, только если она исполняемая** — ложь в ней роняет CI, как ложь в `exports` роняет резолв импорта.

- **По умолчанию: ноль служебных файлов в модуле.** Слой — из положения папки, публичность — из `public/`, зависимости — измеряются линтером по реальным импортам.
- **Манифест — только enforced-override**, меняющий поведение линтера:

```jsonc
// services/legacy-orders/module.json — редкое исключение, не норма
{
  "allowDirectImportFrom": ["checkout"], // временно, до конца миграции
  "deprecated": true, // новые импорты этого модуля запрещены
}
```

Запись, которая пересказывает то, что видно из кода, — запрещена: это документация ради документации.

## Enforcement

Всё, что не проверяется машиной, — не правило, а пожелание, и оно умрёт. Минимальный набор — существующий тулинг:

| Правило                             | Инструмент                                             |
| ----------------------------------- | ------------------------------------------------------ |
| Слои: только «вниз»                 | `eslint-plugin-boundaries`                             |
| Только `*/public/*` снаружи модуля  | `import/no-restricted-paths`                           |
| Запрет barrel (index.ts в модулях)  | `no-restricted-imports` / кастомное правило            |
| Ацикличность графа модулей          | dependency-cruiser `no-circular` (file-scoped линтеру недоступно) |
| Monorepo-стадия                     | Nx `depConstraints`, `exports` в package.json          |

### Стратегия тулинга: единое ядро, а не плагины под каждый линтер

Целевая реализация — **независимый от линтера анализатор**, а не набор плагинов для ESLint/Biome/oxlint. Причины:

1. **Архитектурные правила — свойства графа, а не файла.** Линтеры спроектированы file-scoped: правило видит один файл. Проверка «этот код импортируют трое → кандидат в shared» или поиск циклов требует полного графа импортов. ESLint-плагины границ каждый раз заново резолвят проект внутри чужого жизненного цикла — медленно и хрупко.
2. **Экосистема линтеров фрагментирована.** ESLint — JS-плагины, Biome — GritQL-плагины, oxlint — своя модель. Три реализации одних правил гарантированно разойдутся в семантике. Для подхода, претендующего на стандарт, это смертельно.
3. **Прецеденты подтверждают путь:** dependency-cruiser, knip, Steiger (FSD) — standalone-анализаторы графа, не ESLint-плагины. TypeScript тоже не встраивался в линтеры — есть tsserver, к которому подключаются редакторы.

Архитектура тулинга:

```
ядро: резолв графа импортов + проверка правил (один проход)
│
├── CLI            # source of truth: CI, pre-commit, JSON/SARIF-отчёт
├── LSP-сервер     # мгновенная подсветка в редакторе
└── адаптеры       # тонкие мосты в ESLint/Biome по желанию:
                   # не реализуют правила, а транслируют диагностику ядра
```

Редакторская мгновенность — обязательное требование, а не опция: правило, видимое только в CI, по DX проигрывает правилу, которое подчёркивается при наборе. Поэтому LSP — вторая половина CLI, а не «потом». Адаптеры для линтеров нужны лишь командам, у которых вся диагностика уже идёт через ESLint/Biome-пайплайн, — и они должны оставаться трансляторами отчёта ядра, не самостоятельными реализациями.

Перспектива того же ядра — **architecture server**: детерминированный ответ «куда положить файл», доктор триггеров эволюции (сигналит о переходе стадий), промоушен кодмодом (перенос + переписывание импортов). Дисциплину «правила второго использования» заменяет автоматика. Критерий проектирования: человек **без** тулинга должен мочь жить в проекте, просто медленнее — источник истины один (код), рассинхронизироваться нечему.

## LLM-first

Модуль DMA — готовый контекст-пак для агента: внутренности самодостаточны, контракт — плоский список файлов в `public/`, межмодульные рёбра — обычные импорты, которые анализатор собирает за один проход. «Дай агенту модуль checkout» = модуль + `public/` его прямых соседей по графу, ничего лишнего. Архитектуры, которые нельзя скормить модели по кускам, проигрывают по DX всё сильнее с каждым годом.

## Границы спецификации: что DMA не решает

Честный список того, что осталось за рамками — сознательно или потому что не декларируется в принципе:

- **Нарезка домена.** Самая тяжёлая проблема — cart и checkout это один модуль или два — не решается никакой спецификацией: это моделирование предметной области и структура команд (закон Конвея). DMA умеет *детектировать* плохую нарезку постфактум (много портов, плотный подграф) — детекция не рецепт.
- **Поток данных / state management.** Где живёт клиентский стейт относительно серверного кэша, Flux или сигналы — сознательно вне спеки (иначе прощай фреймворк-агностичность). DMA говорит только *где лежит* стейт (`model/`), не *как устроен*. Совместимо с [Flux](./flux-unidirectional.md) как сквозным принципом.
- **Тестовая стратегия.** Спека фиксирует размещение тестов (colocation), но не пирамиду: что покрывать юнитами, что интеграционно на уровне модуля, что e2e — решение команды.
- **Жизненный цикл API сервисов.** Есть `deprecated`-override, но полного цикла (версионирование public/, окна миграций, обещания совместимости между командами) нет — на стадии C это станет нужным, описано только направление (пакеты + semver).
- **Runtime-границы.** Микрофронтенды, module federation, независимые деплои — вне скоупа; стадии DMA покрывают build-time границы.
- **Рендеринг и роутинг фреймворков.** SSR/RSC/файловый роутинг Next.js — интеграционные вопросы конкретного фреймворка; общий принцип один: роут-файлы фреймворка — тонкие прослойки в `app`, монтирующие `public/` модулей.

## Плюсы

- Один и тот же подход от пет-проекта до monorepo — миграций между архитектурами нет, есть стадии
- «Куда положить файл» — алгоритм, не вкус; предмет споров на ревью исчезает
- Ноль служебных файлов в модуле; один конфиг правил на проект
- Точный граф зависимостей: прямые импорты, никаких barrel-прокладок — быстрый бандл, честный tree-shaking, быстрый TS server
- Плоский `public/` — ранний, физический сигнал распухания контракта
- Межмодульные связи — обычные импорты: go-to-definition работает, роли модулей (сервис/лист) выводятся из графа
- Framework-agnostic: только папки и правила импортов
- Модуль = единица контекста для LLM

## Минусы / ограничения

- **Нет референсной реализации.** Собирается из ESLint-плагинов вручную; architecture server — перспектива, не продукт
- **Нет комьюнити и имени.** FSD даёт общий язык при найме; DMA — нет
- Ацикличность проверяется только графовым анализатором (dependency-cruiser / своё ядро) — file-scoped линтера недостаточно, это ещё один инструмент в пайплайне
- Прямые межмодульные импорты повышают цену удаления «сервисного» модуля (смягчение: граф измерим, вес модуля виден до решения об удалении)
- Публичный файл — конкретный путь: перенос файла из `public/` меняет пути у потребителей (IDE-rename решает, но это операция)
- Переезд модуля `features → services` при появлении входящих рёбер — churn путей, git-истории и, болезненнее всего, **конфликты во всех открытых PR**, задевающих модуль (цена предикатной честности; смягчение — кодмод и переезд отдельным PR без изменений логики); при этом сам момент переезда полезен: принудительный пересмотр API
- Связывание событий в `app/wiring` размазывает сценарий: «что происходит после оформления заказа» видно не из модуля checkout, а из app (компенсация: все связки в одном перечислимом месте, а не по подписчикам всего проекта)
- Сложная доменная логика не покрывается слоями: внутри такого модуля добавляется hexagonal-прослойка (ports/adapters) — решение уровня модуля, не всей архитектуры
- Правило второго использования без тулинга требует дисциплины на ревью
- Не для публикуемых UI-библиотек (там контракт — сам пакет, своя специфика)

## Когда использовать

- Новый продукт с горизонтом роста: старт вдвоём, потенциал до нескольких команд
- Команда устала от споров «entity или feature» и хочет машинных правил вместо конвенций
- Важны производительность бандлера/IDE и честный граф зависимостей
- Активная работа с LLM-агентами над кодовой базой
- Нужна одна архитектура на React/Vue/Svelte/Solid-проекты компании

## Когда НЕ использовать

- Публикуемая UI-библиотека
- Команда не готова настроить и поддерживать ESLint-границы — без enforcement DMA деградирует в обычный feature-based
- Нужен готовый стандарт с комьюнити, доками и наймом «из коробки» → [FSD](./feature-sliced-design.md)
- Прототип на выходные — хватит [Type-based](./type-based.md) / [Page-based](./page-based.md) (хотя стадия 0 DMA — это и есть плоские файлы)

## Антипаттерны

- **Barrel file в модуле** (`index.ts` с реэкспортами) — главный запрет DMA
- Сегментация `public/` подпапками — вместо расщепления модуля
- Реализация публичной входной точки во внутреннем сегменте + пачка 1:1-реэкспортов «для порядка»
- Манифест, пересказывающий то, что видно из кода (слой, список зависимостей)
- Центральный реестр/шина «всех связей приложения» — barrel уровня приложения
- Цикл между модулями, оставленный «на потом» или разорванный `// @ts-ignore`-подобным исключением линтера
- Порт там, где хватило бы прямого импорта (церемония без цикла и без ребра вверх)
- Порт для обхода промоушена: зависимость, легально выражаемая прямым импортом, протащена как runtime-инъекция, чтобы не переезжать в `services/`
- Порт вне `public/ports.ts` — runtime-ребро, которое тулинг не может перечислить
- Вынос в `shared` при первом использовании («пригодится»)
- Сегменты `ui/ model/ api/` на стадии 0–1 (ранний оверхед)
- Модуль в `services/` без входящих рёбер — слой назначен прогнозом («будут переиспользовать»), а не фактом
- Пустая папка `services/`, созданная «на будущее» до первого промоушена
- Правила слоёв «по договорённости», без линтера

## Примеры кода/структуры

### Пример проекта

```
src/
├── app/
│   ├── app.tsx
│   ├── router.tsx              # монтирует */public/*-page
│   └── providers.tsx           # связывание портов (если циклы разрывались)
├── features/                    # листья: никто из модулей их не импортирует
│   ├── checkout/               # модуль стадии 2
│   │   ├── public/
│   │   │   ├── checkout-page.tsx
│   │   │   ├── checkout.events.ts   # события владельца
│   │   │   └── checkout.types.ts
│   │   ├── ui/
│   │   ├── model/
│   │   └── api/
│   └── profile.tsx              # модуль стадии 0 — просто файл
├── services/                    # есть входящие рёбра: стабильный public/
│   └── cart/                    # модуль стадии 1; импортируют checkout и др.
│       ├── public/
│       │   ├── cart-widget.tsx
│       │   └── use-cart.ts
│       └── cart.store.ts
└── shared/
    ├── ui/                      # uikit
    ├── lib/
    └── api/                     # базовый клиент
```

На старте проекта — то же самое, но без `services/`: папка появится при первом промоушене.

### Конфиг границ (весь enforcement проекта)

```js
// eslint: правила слоёв и видимости
'boundaries/element-types': ['error', {
  default: 'disallow',
  rules: [
    { from: 'app',      allow: ['features-public', 'services-public', 'shared'] },
    { from: 'features', allow: ['services-public', 'shared'] },   // ✗ features → features
    { from: 'services', allow: ['services-public', 'shared'] },
    { from: 'shared',   allow: ['shared'] },
  ],
}],
// *-public = <слой>/*/public/** — только это видно снаружи модуля
// + dependency-cruiser: no-circular внутри services (ацикличность графа)
```

### Роутер монтирует модули

```tsx
// app/router.tsx
// Динамический импорт = code-splitting по модулям; для анализатора это такое же ребро
const CheckoutPage = lazy(() => import("@/features/checkout/public/checkout-page"));
// Импорт файла-модуля стадии 0 (features/profile.tsx), не barrel папки
const ProfilePage = lazy(() => import("@/features/profile"));

export const routes = [
  { path: "/checkout", element: <CheckoutPage /> },
  { path: "/profile", element: <ProfilePage /> },
];
```

Границы модулей естественно совпадают с границами чанков: страница-фича грузится лениво целиком, `services` и `shared` попадают в общие чанки по факту использования.

## Связанные архитектуры

- **[Evolution Design (EvoComm)](./evocomm-architecture.md)** — ближайший родственник, скелет слоёв совпадает. Отличия DMA: принадлежность слою — предикат по измеренному графу (не суждение), слой `services` появляется по триггеру (не с первого дня), barrel запрещён, триггеры квантованы
- **[Feature-Sliced Design](./feature-sliced-design.md)** — DMA = «FSD минус entities/widgets/pages (непроверяемые машиной различия), минус index.ts, плюс предикаты и триггеры эволюции»
- **[Modular Monolith](./modular-monolith.md)** — тот же принцип границ; `public/` ≈ `public-api.ts`, но без runtime-файла
- **[Colocation](./colocation.md)** — инвариант №3 DMA
- **[Vertical Slices](./vertical-slices.md)** — модуль DMA = вертикальный слайс
- **[Screaming Architecture](./screaming-architecture.md)** — имена модулей кричат доменом
- **[Monorepo / Packages](./monorepo-packages.md)** — стадия 4
- **[Clean / Hexagonal](./clean-hexagonal.md)** — опциональная прослойка внутри сложного модуля

## Миграция

**Из type-based / page-based:**

1. Создай `app/`, `features/`, `shared/` (`services/` появится при первом промоушене)
2. Каждую страницу/фичу — в модуль стадии 0–1 (файл или папка с `public/`)
3. В `shared` — только то, что реально используют 2+ модуля
4. Включи ESLint-границы сразу — потом дороже

**Из FSD:**

1. `pages` + `widgets` + `features` + `entities` одного домена → один модуль: в `features/`, а если его импортируют другие модули — сразу в `services/`
2. `index.ts` каждого слайса → папка `public/` (реализации внутрь, deep-import снаружи запретить)
3. Cross-import через `@x` → прямые импорты `public/` (проверить ацикличность); найденные циклы разорвать портами
4. `shared` остаётся почти как есть; выкинуть barrel'ы

**Из ED:**

1. Слои `features/` и `services/` остаются; принадлежность проверить предикатом: сервис без входящих рёбер → вернуть в `features`, чистая инфраструктура из `services` → в `shared`
2. `index.ts` модулей → `public/` (реализации внутрь, deep-import снаружи запретить)
3. Правила small/medium → один конфиг границ + dependency-cruiser для ацикличности `services`

## Ключевые термины (для LLM)

| Термин                        | Значение                                                                                   |
| ----------------------------- | ------------------------------------------------------------------------------------------ |
| DMA                           | Derived Modular Architecture — правила выводятся из файловой системы, enforced тулингом    |
| Инвариант                     | Правило, неизменное на любом масштабе (их четыре)                                          |
| Предикат слоя                 | Условие принадлежности слою: «импортируют ли тебя» (машина) × «продуктовый ли сценарий» (суждение) |
| Стадия модуля (0–4)           | Файл → плоский → сегменты → расщепление → пакет                                            |
| Эволюция проекта              | Ленивое появление слоёв: `services/` — первым промоушеном, `domains/*` — при плотном подграфе |
| Триггер                       | Измеримый сигнал перехода на следующую стадию                                              |
| Сегмент                       | Группа по виду кода внутри модуля: `public`, `ui`, `model`, `api`, `lib`                   |
| `public/`                     | Плоская папка входных точек модуля; единственное, что видно снаружи; реализация внутри неё |
| Входная точка                 | Публичный файл модуля; импортируется прямым путём, без barrel                              |
| Barrel                        | `index.ts` с реэкспортами — главный антипаттерн DMA                                        |
| Ацикличность                  | Инвариант графа модулей: циклы запрещены; цикл или ребро вверх — триггер для порта         |
| Сервис                        | Модуль с входящими рёбрами от модулей; живёт в `services/` — по предикату, не по прогнозу  |
| Событие владельца             | Типизированное событие в `public/` эмиттера; подписчик импортирует токен, не реализацию    |
| Порт потребителя              | Интерфейс в `public/ports.ts` потребителя; связывается в `app` при цикле или ребре вверх   |
| Enforced-override             | Единственный допустимый вид манифеста: меняет поведение линтера, ложь роняет CI            |
| Правило второго использования | Подъём кода на уровень выше только при втором использовании                                |
| Architecture server           | Перспективный тулинг: «куда положить файл», доктор триггеров, промоушен кодмодом           |
