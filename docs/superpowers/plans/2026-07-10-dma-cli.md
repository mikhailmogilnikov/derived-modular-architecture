# DMA CLI v1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build `packages/cli` with `dma check` / `dma doctor` over an independent analyzer core for a single-root DMA `src/` tree.

**Architecture:** Pure `core` pipeline (`discover → parse → graph → classify → rules/signals`) returns diagnostics; thin `cli` owns argv, formatters, and exit codes. Zero-config: only `src/{app,features,services?,shared}`.

**Tech Stack:** Bun workspaces, TypeScript, TypeScript 5.x compiler API (`createSourceFile`) for import parsing inside `packages/cli`, Node `util.parseArgs` for CLI, bun:test for tests. No ESLint/cruiser dependency.

**Parser dependency note:** Root monorepo may keep TypeScript 7 for tooling, but `@derived-modular/cli` depends on `typescript@5.9.x` because TS 7's `typescript/unstable/sync` API requires Node's `stdout._handle.fd` and crashes under Bun (`TypeError: undefined is not an object (evaluating 'stdout._handle.fd')`). Do not use TS 7 sync/async project APIs for parsing in this package.

**Spec:** `docs/superpowers/specs/2026-07-10-dma-cli-design.md`

## Global Constraints

- Core must never write to stdout/stderr or call `process.exit`.
- `import type` counts as a graph edge.
- Inbound edges for predicates = imports from other modules (`features/*` | `services/*`) only; `app/` mounts do not promote.
- Exit codes: `0` ok, `1` check errors, `2` environment failure.
- No config file, no autofix, no monorepo/`domains/*` in v1.
- Package manager: bun. Lint/format via ultracite after edits when convenient.
- Root `package.json` workspaces must include `packages/*`.

Logic in `discover.ts`:

### Task 5: Check rules

**Interfaces:**
- Consumes: discovered + graph + classified
- Produces: `runCheckRules(...): Diagnostic[]` with ruleIds from spec

Layer rank: `app=3, features=2, services=1, shared=0`. Illegal if importer rank < imported rank (upward), or equal ranks that are both `features` (feature-to-feature). `shared→shared` allowed. `services→services` allowed (cycles checked separately).

Rules to implement exactly:
1. `layer-direction` — upward import between layers
2. `feature-to-feature` — module edge features→features
3. `public-api` — cross-module file edge where target is inside a dir module but path is not under that module's `public/`, and target is not a stage-0 file module root
4. `no-barrel` — module contains `index.ts`/`index.tsx` that has any `export ... from` (re-export). Flag the barrel file.
5. `no-cycle` — SCC size > 1 in module graph (features + services)
6. `feature-has-inbound` — features module with inboundFromModules size > 0
7. `service-no-inbound` — services module with inbound size === 0

Each diagnostic: severity `error`, `file` + `range` when from an import edge, `help` with short remediation.
